## 3. Branching & release strategy

- `main` stays releasable. No long-lived feature branches.
- Each task = feature branch `v1.1/<task-id>-<slug>`, reviewed via two-stage
  review (spec compliance → code quality per `subagent-driven-development`),
  then squashed into main.
- Balance-ops schema migration is non-reversible once users upgrade. Gate it
  behind a one-time backup-first prompt on first v1.1 launch.
- Weekly internal "dogfood" tag: `v1.1.0-wk1`, `v1.1.0-wk2`, etc., so the user
  can test partial builds.

## 7. Per-task subagent dispatch template

Every task below is dispatched to a fresh implementer subagent via
`superpowers:subagent-driven-development`. The controller (me) feeds each
subagent the full task text + context; subagent asks clarifying questions if
needed; subagent implements + tests + self-reviews + commits; controller
dispatches spec-reviewer then code-quality-reviewer; any issues → same subagent
fixes → re-review; task marked complete.

**Dispatch template for each task:**

```
Task: T{X.Y} — {title}
Scenarios: {S## refs}

  - CLAUDE.md (hard rules)

  - {list of prior tasks}

  - {file-list + test-list + doc-list}
Exit criteria:
  - npm test green
  - npm run typecheck green
  - Each scenario in {S##} provably works (demo script or test)

```

Each task's full dispatch brief will be written at time of dispatch, not
pre-authored, so the subagent gets the *current* state of main including
whatever the prior task changed.

## 9. Data integrity / migration plan

v1.1.0 introduces breaking schema changes. Migration strategy:

1. On first v1.1 launch, app checks DB `user_version` pragma.
2. If < v1.1 schema version: prompt "We're about to upgrade your data. Ledger
   will back up first. Continue?" Cancel exits app.
3. Backup written to `%APPDATA%\FXLedger\backups\pre-v1.1-YYYYMMDD-HHMMSS.zip`.
4. Drizzle migration `0003_v1_1_balance_ops.sql` runs atomically in a
   transaction with `PRAGMA user_version = X`.
5. Failure → transaction rolls back, app shows error + restore instructions,
   exits.
6. Success → app continues to new version.
