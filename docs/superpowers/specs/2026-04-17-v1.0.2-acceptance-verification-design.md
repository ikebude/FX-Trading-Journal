## 1. Goal

Ship Ledger v1.0.2 as the **first release that has systematically passed all 24 acceptance criteria** defined in [`PROJECT_BRIEF.md` §9](../../../PROJECT_BRIEF.md). Until this release, the app compiles and runs but has never been verified end-to-end against the acceptance checklist that defines "done" for the product.

v1.0.2 is a **verification + polish release** that also lights up auto-update:

1. Fills the remaining help-UI gaps (module 17c) so acceptance criterion #23 can pass.
2. Makes auto-update actually shippable — IPC plumbing, in-app update banner, manual check button. The config flag and default-off toggle already exist per Hard Rule #11; what ships here is the user-facing UX.
3. Adds an automated E2E test suite covering the 18 automatable criteria.
4. Adds a manual test playbook covering the 6 criteria that require Windows-specific interactions (install flow, global hotkey, tray, cross-machine backup, OneDrive data folder) + one new Update flow sign-off.
5. Fixes any bugs surfaced during verification.
6. Updates CLAUDE.md to reflect current reality (current "Remaining" list is stale).

**Exit gate:** all 24 acceptance criteria pass — 18 via automated tests, 6 via signed-off manual playbook — on a clean Windows install of `Ledger Setup 1.0.2.exe`.

## 3. Scope

### 3.1 In scope

- Build the three 17c help surfaces.
- Ship end-to-end auto-update UX (IPC plumbing + UpdateBanner + manual check in Settings).
- Write a Playwright+Electron E2E suite covering 18 of 24 acceptance criteria.
- Write a manual test playbook (`docs/acceptance-test-playbook.md`) for the remaining 6 criteria + 1 new Update flow sign-off.
- Run the E2E suite + playbook; fix any bugs surfaced.
- Update CLAUDE.md to reflect reality.
- Ship v1.0.2 via the existing GitHub Actions release workflow.

### 3.2 Out of scope

- New business-logic features (nothing in PROJECT_BRIEF §6 is missing).
- Performance work beyond what acceptance criteria require (e.g., AC-24 requires a visible progress bar during a 500-row import — the bar exists; we only verify it).
- CI-based Playwright run. The E2E suite is developer-run locally; the existing `ci.yml` workflow is not extended in this release (Electron on CI windows-latest adds 5+ min per run and produces flaky native-module installs — deferred).
- Automated E2E coverage of the auto-update flow. `electron-updater` would need a mock GitHub Releases server to test end-to-end, which is brittle. The flow is verified manually via the playbook — once post-v1.0.2.

### Phase 2 — Auto-update UX

Goal: make the already-scaffolded `electron-updater` integration actually actionable from inside the app. Today the config flag and Settings toggle exist; this phase adds the IPC plumbing, in-app banner, manual check button, and restart-to-install flow.

**Architectural split:** main-process driven, default OFF per Hard Rule #11. The renderer never talks to GitHub directly — all network happens in `electron-updater` running in main, forwarded to the renderer via IPC events.

**New files:**

| File | Purpose | ~LOC |
|---|---|---|
| `electron/services/auto-update.ts` | Extracts `autoUpdater` wiring out of `main.ts` into its own service. Registers event listeners (`checking-for-update`, `update-available`, `update-not-available`, `download-progress`, `update-downloaded`, `error`), forwards each to the main window via `webContents.send('updater:event', …)`. Enforces a 4-hour cooldown between auto-checks. | ~150 |
| `electron/ipc/updater.ts` | IPC handlers for `updater:check` (manual trigger), `updater:download` (user consent to download), `updater:install` (restart and apply). | ~80 |
| `src/components/layout/UpdateBanner.tsx` | Yellow strip rendered above `<main>` in `AppShell`. Shows "Ledger v{newVersion} is available — Download • Dismiss"; becomes a progress bar during download; becomes "Update ready — Restart now" when complete. Dismissal is session-local (reappears next launch). | ~180 |
| `src/hooks/useUpdater.ts` | Subscribes to `window.ledger.updater.onEvent`, owns updater state `{ status, version, progress, error, lastCheckedAt }`, exposes `check()`, `download()`, `install()`. | ~100 |

**Files to modify:**

| File | Change |
|---|---|
| `electron/main.ts` | Remove the inline `autoUpdater.checkForUpdatesAndNotify()` call; delegate to `runAutoUpdateCheck()` from the new service. |
| `electron/preload.ts` | Add `ledger.updater = { check, download, installAndRestart, onEvent }`. |
| `electron/ipc/index.ts` | Register the new `updater` handlers. |
| `src/App.tsx` | Render `<UpdateBanner />` inside `AppShell`, below `<TopBar />` and above `<PropFirmBanner />`. |
| `src/pages/SettingsPage.tsx` | Add "Check for updates now" button next to the auto-update toggle. Button works regardless of toggle state. Shows "Last checked: {relative time}" underneath. |

**IPC surface (final):**

```ts
window.ledger.updater = {
  check(): Promise<CheckResult>                 // {status: 'checking'|'up-to-date'|'available'|'error', version?, error?}
  download(): Promise<void>                     // user consent; no-op if no update
  installAndRestart(): void                     // fires-and-forgets; app quits mid-call
  onEvent(cb: (e: UpdaterEvent) => void): () => void   // returns unsubscribe
}
```

**Event shape forwarded to renderer:**

```ts
type UpdaterEvent =
  | { type: 'checking' }
  | { type: 'available'; version: string; releaseNotes?: string }
  | { type: 'up-to-date'; version: string }
  | { type: 'progress'; percent: number; bytesPerSecond: number; transferred: number; total: number }
  | { type: 'downloaded'; version: string }
  | { type: 'error'; message: string }
```

**Behavior:**

- App startup: if `auto_update === true` and more than 4 hours since last check, `autoUpdater.checkForUpdates()` fires in the background. No UI unless an update is found.
- Manual button in Settings: always works, ignores cooldown.
- `autoDownload = false`: download never starts until the user clicks Download in the banner.
- `autoInstallOnAppQuit = false`: install only happens after explicit "Restart now" click.
- Errors are surfaced in the banner as a red variant, not hidden.

**Phase 2 done when:**
- Flipping `auto_update` on and clicking "Check for updates now" produces a banner (or "up-to-date" toast) within ~5 s.
- With the toggle off, the manual check still works.
- A mock-server dry run (developer-local, using `nock` or `updater.setFeedURL` against a local file) walks through the full flow: check → available → download → progress → ready → restart.
- `npm test`, `npm run typecheck`, `npm run lint` all pass.

**`helpers.ts` API:**

```ts
launchApp(dataDir?: string): Promise<{ app: ElectronApplication; window: Page; cleanup: () => Promise<void> }>
seedAccount(window: Page, opts: { type: 'PROP' | 'LIVE'; balance: number; propRules?: PropRules }): Promise<string>
seedTrades(window: Page, count: number, opts?: { accountId: string; symbol?: string }): Promise<void>
importStatement(window: Page, fixturePath: string): Promise<{ imported: number; failed: number }>

waitForToast(window: Page, substring: string, timeoutMs?: number): Promise<void>
waitForRoute(window: Page, path: string, timeoutMs?: number): Promise<void>
```

**Each test launches its own Electron instance with a temp data directory** (via `os.tmpdir() + randomUUID()`) so tests never pollute `%APPDATA%\Ledger`. `cleanup()` closes the app and removes the temp dir.

## 5. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Playwright+Electron flake on Radix tooltip hover timing | Use `expect(...).toBeVisible({ timeout: 2000 })` with explicit waits; avoid `.hover()` race by using `page.locator(...).hover()` followed by `waitFor({ state: 'visible' })` |
| E2E tests pollute dev data folder | Every test gets a unique `os.tmpdir()/ledger-e2e-<uuid>` dir; `cleanup()` always runs in `afterAll`/`afterEach` |
| Native module rebuild fails on fresh Electron version | Existing v1.0.1 release build proves this works; no Electron upgrade in v1.0.2 |
| Tooltip insertion creates a huge diff across blotter + dashboard | Keep `MetricTooltip` as a pure wrapper — no styling changes; diff is N insertions, not N rewrites |
| A criterion fails and requires deep code change | If the fix is small, land it in the same commit; if architectural, file an issue and defer to v1.0.3, documenting the failure in CHANGELOG |
| Auto-update check against `github.com/ikebude/FX-Trading-Journal/releases/latest.yml` fails because v1.0.1 didn't publish a `latest.yml` | Release workflow was fixed in commit `072ad97` to upload `latest.yml`. v1.0.2 is the first release that ships with it; v1.0.1-installed users with auto-update enabled will detect v1.0.2 correctly. |
| User clicks "Check for updates" on v1.0.2 before v1.0.3 exists | Expected and correct — banner shows "up-to-date". Not a bug. |
| `autoUpdater` crashes on a malformed `latest.yml` | `autoUpdater.on('error', …)` forwards the error to the renderer as a banner with red variant; app keeps running. Never crashes the window. |
