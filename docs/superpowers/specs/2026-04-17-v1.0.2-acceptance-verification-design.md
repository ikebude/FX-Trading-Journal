## 1. Goal

Ship Ledger v1.0.2 as the **first release that has systematically passed all 24 acceptance criteria** defined in [`PROJECT_BRIEF.md` §9](../../../PROJECT_BRIEF.md). Until this release, the app compiles and runs but has never been verified end-to-end against the acceptance checklist that defines "done" for the product.

v1.0.2 is not a new-feature release. It is a **verification + polish release** that:

1. Fills the remaining help-UI gaps (module 17c) so acceptance criterion #23 can pass.
2. Adds an automated E2E test suite covering the 18 automatable criteria.
3. Adds a manual test playbook covering the 6 criteria that require Windows-specific interactions (install flow, global hotkey, tray, cross-machine backup, OneDrive data folder).
4. Fixes any bugs surfaced during verification.
5. Updates CLAUDE.md to reflect current reality (current "Remaining" list is stale).

**Exit gate:** all 24 acceptance criteria pass — 18 via automated tests, 6 via signed-off manual playbook — on a clean Windows install of `Ledger Setup 1.0.2.exe`.

## 3. Scope

### 3.1 In scope

- Build the three 17c help surfaces.
- Write a Playwright+Electron E2E suite covering 18 of 24 acceptance criteria.
- Write a manual test playbook (`docs/acceptance-test-playbook.md`) for the remaining 6 criteria.
- Run the E2E suite + playbook; fix any bugs surfaced.
- Update CLAUDE.md to reflect reality.
- Ship v1.0.2 via the existing GitHub Actions release workflow.

### 3.2 Out of scope

- New business-logic features (nothing in PROJECT_BRIEF §6 is missing).
- Performance work beyond what acceptance criteria require (e.g., AC-24 requires a visible progress bar during a 500-row import — the bar exists; we only verify it).
- CI-based Playwright run. The E2E suite is developer-run locally; the existing `ci.yml` workflow is not extended in this release (Electron on CI windows-latest adds 5+ min per run and produces flaky native-module installs — deferred).
- Auto-update end-to-end (electron-updater is off by default per Hard Rule #11; `latest.yml` is published for future opt-in).

### Phase 2 — Acceptance E2E Suite

Goal: automate 18 of 24 acceptance criteria via Playwright+Electron.

**New files:**

| File | Purpose |
|---|---|
| `tests/e2e/acceptance-criteria.spec.ts` | One `test()` block per automatable criterion, named `AC-02`, `AC-04`, …, `AC-24` |
| `tests/e2e/helpers.ts` | Shared utilities: `launchApp`, `seedAccount`, `seedTrades`, `importStatement`, `dropBridgeEvent`, `waitForToast`, `waitForRoute` |
| `tests/e2e/fixtures/mt4-sample.htm` | MT4 detailed statement — ~30 trades, mix of open/closed, includes partial closes |
| `tests/e2e/fixtures/mt5-sample.htm` | MT5 detailed statement — ~30 trades across EURUSD, USDJPY, XAUUSD |
| `tests/e2e/fixtures/generic-broker.csv` | CSV with non-standard column order (tests fuzzy header matcher) |
| `tests/e2e/fixtures/forexfactory.csv` | ForexFactory news calendar export sample |
| `tests/e2e/fixtures/bridge-event-open.json` | Simulated MT5 trade-open event |
| `tests/e2e/fixtures/bridge-event-close.json` | Simulated MT5 trade-close event |

**`helpers.ts` API:**

```ts
launchApp(dataDir?: string): Promise<{ app: ElectronApplication; window: Page; cleanup: () => Promise<void> }>
seedAccount(window: Page, opts: { type: 'PROP' | 'LIVE'; balance: number; propRules?: PropRules }): Promise<string>
seedTrades(window: Page, count: number, opts?: { accountId: string; symbol?: string }): Promise<void>
importStatement(window: Page, fixturePath: string): Promise<{ imported: number; failed: number }>
dropBridgeEvent(dataDir: string, fixturePath: string): Promise<void>
waitForToast(window: Page, substring: string, timeoutMs?: number): Promise<void>
waitForRoute(window: Page, path: string, timeoutMs?: number): Promise<void>
```

**Each test launches its own Electron instance with a temp data directory** (via `os.tmpdir() + randomUUID()`) so tests never pollute `%APPDATA%\Ledger`. `cleanup()` closes the app and removes the temp dir.

## 5. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Playwright+Electron flake on Radix tooltip hover timing | Use `expect(...).toBeVisible({ timeout: 2000 })` with explicit waits; avoid `.hover()` race by using `page.locator(...).hover()` followed by `waitFor({ state: 'visible' })` |
| E2E tests pollute dev data folder | Every test gets a unique `os.tmpdir()/ledger-e2e-<uuid>` dir; `cleanup()` always runs in `afterAll`/`afterEach` |
| Native module rebuild fails on fresh Electron version | Existing v1.0.1 release build proves this works; no Electron upgrade in v1.0.2 |
| Tooltip insertion creates a huge diff across blotter + dashboard | Keep `MetricTooltip` as a pure wrapper — no styling changes; diff is N insertions, not N rewrites |
| A criterion fails and requires deep code change | If the fix is small, land it in the same commit; if architectural, file an issue and defer to v1.0.3, documenting the failure in CHANGELOG |
