# Ledger v1.0.2 — Multi-Hat Audit

**Date:** 2026-04-18
**Companion to:** [2026-04-18-v1.1-product-audit-design.md](2026-04-18-v1.1-product-audit-design.md)
**Purpose:** Every senior role on the team conducts an independent audit of v1.0.2. Findings are consolidated, de-duplicated, graded, and either (a) folded into the v1.1 scope if truly P0/P1, or (b) parked in the long-range roadmap. **User approval required** before the v1.1 scope changes.

**Severity grades:**
- **P0** — ship-blocker (security, data loss, real-money usability gap)
- **P1** — major UX/workflow hole, should be in v1.1
- **P2** — polish/best-in-class gap, stage in v1.2-v1.3
- **P3** — strategic/nice-to-have, v2.0 or later

Every finding lists a file path or concrete evidence. No speculation-only items.

---

## Role 4 — Security Engineer / Application Security

*Persona: former bug-bounty hunter, thinks every input is hostile.*

| # | Finding | Evidence | Severity | Proposed |
|---|---|---|---|---|
| **S1** | **Zip-slip vulnerability in backup restore.** [electron/ipc/backup.ts:326-330](electron/ipc/backup.ts#L326-L330) constructs `dest = join(stageDir, name)` where `name` is the ZIP entry filename. An entry named `../../../../Windows/System32/evil.dll` will traverse out of `stageDir`. Node's `path.join` normalizes but does **not** enforce containment. Attack scenario: user restores a malicious "backup.zip" received by phishing. | [backup.ts:326-330](electron/ipc/backup.ts#L326-L330) | **P0 — SECURITY** | v1.1 — validate every entry: `const resolved = resolve(dest); if (!resolved.startsWith(resolve(stageDir) + sep)) throw`. Also reject absolute paths and `..` segments before even joining. |
| **S2** | **No CSP on file://loaded index.html in production.** Meta-tag CSP exists ([index.html:12-14](index.html#L12-L14)) but misses `connect-src`, `frame-ancestors`, `base-uri`, `form-action`, `object-src`. Also uses `'unsafe-inline'` for scripts in production (not only dev). | [index.html:13](index.html#L13) | **P1** | v1.1 — tighten CSP: `connect-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'none'; object-src 'none';` — remove `'unsafe-inline'` from scripts in prod build (Vite can do nonce-based) |
| **S3** | **No `setPermissionRequestHandler` in main.** Renderer could request geolocation / media / notifications — currently Electron grants by default. | No such call in [electron/main.ts](electron/main.ts) | **P1** | v1.1 — `session.defaultSession.setPermissionRequestHandler((_, __, cb) => cb(false))` — deny all; whitelist if a specific permission is legitimately needed |
| **S4** | **Preload bridge exposes whole `ledger` namespace without channel-level auth.** Any code running in renderer (XSS victim or malicious ext) can call `window.ledger.trades.hardDelete()`. | [electron/preload.ts](electron/preload.ts) | **P2** | v1.2 — low priority since renderer has no network access and CSP blocks external scripts. Document the threat model explicitly. |
| **S5** | **Screenshot import does not strip EXIF.** A phone-captured MT4 screenshot carries GPS coordinates. A shared screenshot via CSV export leaks location. | [sharp](https://sharp.pixelplumbing.com/) is already a dep; must verify EXIF stripping default | **P1** | v1.1 — sharp defaults to stripping; verify + add explicit `.rotate().withMetadata(false)` pipeline; add a test |
| **S6** | **Auto-update integrity verification.** electron-updater verifies signatures *only if* the installer is code-signed. Since the project intentionally ships unsigned, there's no chain-of-trust on auto-update. A MITM on the GitHub fetch could substitute a malicious installer. | Not signed per [CLAUDE.md](CLAUDE.md) | **P1** | v1.1 — add a SHA-256 hash published separately (e.g. in the release body) that users can verify manually; document in README. Long-term: code signing (v2.0). |
| **S7** | **Log files could leak PII despite rule #12.** `electron-log` at `info` level — need to audit every `log.info(...)` for account names, symbols, dollar amounts. | [CLAUDE.md](CLAUDE.md) rule 12 says no, but needs enforcement | **P1** | v1.1 — grep `log\.(info\|warn\|error)` and audit every call; add lint rule to forbid interpolating specific field names |
| **S8** | **No lockfile integrity on startup.** If someone tampers with the DB file directly, Ledger opens it silently. Could incorporate a per-row SHA check or a rolling HMAC. | No check | **P3** | v2.0 — optional "tamper-evident mode" using HMAC chain |
| **S9** | **Settings → Move data folder could accept UNC path / symlink loops.** Pathological inputs could hang migration. | [electron/ipc/settings.ts](electron/ipc) | **P2** | v1.2 — validate: must be writable, must not be on network drive unless explicitly allowed, cannot be ancestor or descendant of current data dir |
| **S10** | **IPC handlers may not universally Zod-validate input.** Rule: every IPC handler validates via [src/lib/schemas.ts](src/lib/schemas.ts). | Needs audit | **P1** | v1.1 — sweep every `ipcMain.handle` — confirm Zod or equivalent; add missing |
