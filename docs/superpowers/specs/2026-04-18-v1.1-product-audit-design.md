# Ledger v1.1 — Product Audit & Roadmap

**Date:** 2026-04-18
**Baseline:** v1.0.2 (shipped, see [CHANGELOG.md](../../../CHANGELOG.md))
**Audit source:** User hands-on test-run of installed v1.0.2 + code-level verification
**Purpose:** Capture every real gap between Ledger v1.0.2 and best-in-class forex journals, prioritize a v1.1 release, and stage v1.2 / v1.3 / v2.0.

---

## 1. Executive summary

v1.0.2 ships the **technical foundation**: P&L engine, 18-table schema, three importers, live MT4/5 bridge, auto-update, system tray, prop-firm guardrails, and 24 acceptance criteria. What it does *not* yet ship is the **workflow surface** traders need day-to-day:

- **Account creation UI is entirely missing.** The backend supports LIVE / DEMO / PROP account types and a `broker` field, but there is no way to create a new account from the running app. Users are stuck with the single seeded DEMO account. This is the single most painful gap — it blocks everyone from using Ledger for real money.
- **TradeForm mixes data-typed fields with free-text fields inconsistently.** `Symbol` and `Setup` are both free text. The `instruments` table exists and is populated but is never used for autocomplete; `setup_name` is stored but never reused.
- **Broker context is not modeled.** A live account has a broker name, server, account number, base currency, leverage, platform (MT4/MT5/cTrader/TradingView), regulatory region, and account opening date — Ledger stores only `broker` as free text.
- **No trading playbook layer.** Edgewonk, TradeZella, and TraderVue all treat *named strategies with expected stats* as first-class entities. Ledger stores `setup_name` as a denormalized string with no aggregation schema.
- **No mistake / tilt modeling.** The schema captures `preTradeEmotion` / `postTradeEmotion` enums but not the trading-mistake vocabulary that differentiates journals designed for *improvement* from journals designed for *record-keeping*.

These gaps are not architectural — the schema is flexible enough to support v1.1 without migrations. This is **product work**, not platform work.

MyFxBook is social/verification, not journaling. Prop-firm dashboards (FTMO, MFF) are compliance, not analysis.

**Positioning insight:** Ledger's three genuine differentiators are (1) **local-first privacy**, (2) **live MT4/5 bridge** (most competitors only import statements after the fact), and (3) **prop-firm guardrails with IANA timezone-aware daily reset**. Every v1.1 decision should protect those three moats — not chase feature-parity for its own sake. If a competitor feature requires a cloud service or compromises the offline-first story, it's *not* a roadmap item.

## 4. v1.1 scope — blocker-level UX fixes

**Ship target:** 10–14 days of focused work. Closes every P0 and the highest P1s. No schema-invasive changes (existing tables gain columns; no table renames, no enum changes that break migrations).

### 4.1 — Account creation & broker metadata (G1, G3)

**Schema migration** (additive, backward-compatible):

```ts
// src/lib/db/schema.ts — accounts table additions
server: text('server'),                                     // MT server URL / name
externalAccountNumber: text('external_account_number'),     // MT login id
platform: text('platform', {                                // trading platform
  enum: ['MT4', 'MT5', 'CTRADER', 'TRADINGVIEW', 'NINJATRADER', 'OTHER']

leverage: integer('leverage'),                              // e.g. 500 for 1:500
```

**New component:** [src/components/account-form/AccountDialog.tsx](src/components/account-form/AccountDialog.tsx) (to be created)

- Radix Dialog — accessible from:
  - Top-bar `+` button next to [AccountSelector](src/components/layout/AccountSelector.tsx)

Replace [TradeForm.tsx:213-218](src/components/trade-form/TradeForm.tsx#L213-L218) `<Input placeholder="EURUSD">` with a **combobox** backed by the `instruments` table.

Replace [TradeForm.tsx:275-277](src/components/trade-form/TradeForm.tsx#L275-L277) free `<Input>` with a combobox backed by `SELECT DISTINCT setup_name FROM trades WHERE setup_name IS NOT NULL ORDER BY COUNT(*) DESC`.

### 4.4 — Mistake tagging (G7)

**Schema:**
```ts
// new table
export const tradeMistakes = sqliteTable('trade_mistakes', {
  tradeId: text('trade_id').references(() => trades.id, { onDelete: 'cascade' }),
  mistake: text('mistake', {
    enum: ['REVENGE', 'FOMO', 'OVERSIZED', 'MOVED_STOP', 'NO_STOP',
           'BROKE_RULE', 'CHASED', 'HESITATED', 'PREMATURE_EXIT',
           'HELD_TOO_LONG', 'OVERTRADED']
  }).notNull(),
}, (t) => ({
  pk: primaryKey({ columns: [t.tradeId, t.mistake] }),
}));
```

**UI:** Add a "Mistakes" chip row to TradeForm's **Context** tab (next to emotion chips). Multi-select. Each chip has a tooltip explaining the category. Dashboard gets a new widget: "Top 3 mistakes by P&L impact (last 30 trades)".

Use React Hook Form's `useWatch` so the preview re-renders without causing full form re-renders.

### 4.6 — Primary "+ New trade" CTA (G5)

### 4.7 — Import/account sanity check (G6)

When user picks an account in [ImporterPage.tsx:509](src/pages/ImporterPage.tsx#L509), run a lightweight signature check on the uploaded statement:

## 11. Decision log — what we're deferring and why

- **Chart annotation, trade replay, playbook entities:** real UX wins but 2+ weeks each. Don't block v1.1 behind them.
- **Monte Carlo / correlation / what-if analytics:** differentiator material but most traders won't touch these until they have consistent data flow. Ship after v1.1 makes data entry frictionless.
- **Mobile companion:** requires a sync story (OneDrive / user S3) that must not compromise offline-first. Design, don't build, in v1.1.
- **Local AI pattern detection:** pure moat feature. Ollama binary is 500MB+ — installer weight doubles. Make it an optional "Install AI pack" in v2.0, not part of the core installer.
- **Code signing:** the $300-ish/yr EV cert cost is a subscription lock-in the project was designed to avoid. Revisit when traction justifies it.
- **Cloud sync with Ledger-operated server:** never. Breaks the zero-telemetry story that differentiates us.

## Appendix A — Why we should *not* chase feature parity

Edgewonk, TradeZella, and TraderVue are all cloud-hosted subscription products. Their feature sprawl is monetization-driven: every new checkbox is a reason to renew.

Ledger's value proposition is the opposite: **what does a serious trader actually need, offline, forever, free.** That means a ruthless "no" to:

- Social sharing / leaderboards (privacy-hostile)
- Trade-idea marketplaces (content-quality rabbit hole)
- Automated broker connections for non-MT platforms that require OAuth to a cloud service (breaks local-first)
- Any feature that *only* works with internet connectivity for day-to-day use

The competitive benchmark is **"everything in Edgewonk/TradeZella that a pro needs, minus the subscription, minus the cloud"** — not "everything they have." This document's prioritization reflects that.
