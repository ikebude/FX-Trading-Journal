# FXLedger v1.1 — Path-to-Assurance Roadmap

This document tracks the concrete deliverables required to raise end-to-end
assurance from "stable for journaling + imports + analytics" (v1.0.7) to
"a 5+ year forex trader can run the app end-to-end in every area without
encountering errors, bugs or build gaps" (v1.1 target).

Each item below is scoped, testable, and has a clear "done" signal.
Items marked `[IN PROGRESS]` have partial implementation; items marked
`[PLANNED]` have a clear design but no code yet.

---

## R1 — Acceptance E2E suite realignment `[IN PROGRESS]`

**File:** `tests/e2e/acceptance-criteria.spec.ts` (25 tests, currently in
`testIgnore` in `playwright.config.ts`).

**Problem:** The suite was authored against the pre-v1.0.6 UI. Specific
misalignments:
- Routes: uses `/import`, `/blotter/new` — current UI uses `/importer`,
  `/trades/new`.
- Zod payload shapes: older field names (`entry_ts` vs. `openedAtUtc`,
  `symbol` vs. `instrumentId`).
- Seed fixtures assume direct SQLite writes — current fixtures must go
  through IPC handlers to respect audit-log invariants.

**Done signal:**
- [ ] All 25 tests realigned and passing against current routes/schemas.
- [ ] `testIgnore` entry removed from `playwright.config.ts`.
- [ ] CI `test` job runs `npm run test:e2e` after unit tests.

**Estimate:** 3–5 eng-days.

## R2 — Broker CSV dialect fixtures `[PLANNED]`

**Problem:** Current parser tests cover the "generic" CSV + MT4/MT5 HTML
shapes. Real-world traders export from many platforms with different
delimiters, encodings, decimal separators and column orders.

**Target fixtures** (one minimal CSV + one expected-parsed JSON per broker):
- [ ] cTrader (comma-separated, UTC ISO timestamps)
- [ ] IC Markets MT4 (tab-separated, broker-local timestamps)
- [ ] Pepperstone MT5 (comma-separated, millisecond precision)
- [ ] Exness MT4 (semicolon-separated, EU decimal comma)
- [ ] OANDA v20 (comma-separated, ISO timestamps with TZ offset)

**Implementation:**
- `tests/fixtures/broker-csv/{broker}.csv` + `{broker}.expected.json`
- `tests/imports.broker-dialects.test.ts` — parametrized over the fixture
  directory.

**Done signal:**
- [ ] All five fixtures parse without error.
- [ ] Parsed trades match expected JSON (timestamps, prices, lots, P&L).
- [ ] Unknown symbols in fixtures surface as a clear "register instrument"
      prompt rather than silent 0-P&L trades.

**Estimate:** 2–3 eng-days.

## R3 — E2E coverage for remaining user journeys `[PLANNED]`

**Gaps not covered by current smoke (`launch + dashboard + settings`):**
- [ ] Import round-trip: drop CSV → preview → commit → trades visible in
      blotter → audit_log rows created.
- [ ] PDF/CSV export: generate report → file exists on disk → non-empty.
- [ ] Backup round-trip: create backup zip → wipe data → restore → trades
      match.
- [ ] Auto-updater: staged local update server → in-app check detects
      version → download → install-and-restart (mocked).
- [ ] EA bridge handshake: start mock EA server → app receives trade →
      visible in blotter within SLA.
- [ ] Capture overlay + hotkey screenshot: overlay shows → capture →
      screenshot attached to open trade.

**Done signal:**
- [ ] Six new Playwright specs under `tests/e2e/journeys/`.
- [ ] All pass under `npm run test:e2e` on CI.

**Estimate:** 5–7 eng-days.

## R4 — First-run onboarding wizard `[PLANNED]`

**Problem:** A senior trader on first launch must understand:
- Data folder location and why it cannot be changed lightly.
- Account metadata (broker server, platform, leverage, timezone, login,
  broker type) and why correct timezone matters for P&L attribution.
- Instrument registration (`pip_size`) — unknown symbols break pip math.

**Implementation:**
- `src/pages/OnboardingPage.tsx` — 4-step wizard (data folder → account →
  instruments → calendar preferences).
- Triggered from `useEffect` in `App.tsx` when `settings.onboardingComplete`
  is false.
- Persisted flag in settings table.

**Done signal:**
- [ ] Wizard appears on fresh install.
- [ ] User cannot dismiss until each step passes validation.
- [ ] E2E test covers the happy path.

**Estimate:** 3–4 eng-days.

**Implemented:** `.github/workflows/nightly.yml` runs on a cron schedule

**Done signal:** ✅ Workflow file present; first green run required before
closing R5.

## R6 — Dual-ABI native module strategy `[DONE]`

**Implemented in v1.0.7 release hardening:**
- `package.json` exposes `rebuild:node` and `rebuild:electron` scripts.
- `ci.yml` swaps to Node-ABI prebuilt before `npm test`.
- `release.yml` swaps to Node ABI before tests, then to Electron ABI before
  packaging.
- `postinstall` remains `electron-rebuild` so `npm run dev` works out of the
  box after a fresh `npm ci`.

**Done signal:** ✅ Fixed the 2026-04-21 GitHub Actions failure on `main`
(`NODE_MODULE_VERSION 145` vs `115` mismatch). Verified locally on
Windows + Node 24.15.0.

---

## R7 — Release gate tightening `[PLANNED]`

Once R1–R4 land, the release gate in `release.yml` must be extended:

```yaml
- name: Run full E2E acceptance suite
  run: npm run test:e2e
```

and `testIgnore` removed from `playwright.config.ts`. Only then can the
v1.1 release notes honestly claim "full end-to-end acceptance verified".
